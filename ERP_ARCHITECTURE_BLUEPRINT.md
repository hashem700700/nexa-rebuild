# وثيقة ميثاق وبنية النظام المعمارية الحاكمة (Enterprise ERP Architecture Blueprint)
### ميثاق الحوكمة وهندسة الأنظمة متعددة الشركات ومستودعات البيانات المالية والتشغيلية

---

## مقدمة وميثاق التنفيذ التدريجي الإلزامي (Execution Governance Contract)

النظام مصمم كـ **Modular Monolith** مالي وتشغيلي عملاق متعدد الشركات والشركاء (Multi-Tenant)، يعتمد على قاعدة بيانات **PostgreSQL** كمصدر وحيد وبات وغير قابل للنقض للحقيقة (**Single Source of Truth**).

### ميثاق الترتيب والتعاقب الصارم للتنفيذ (Mandatory Sequencing Order)
يُحظر تمامًا وبشكل قاطع البدء في تصميم أو تطوير أي واجهة مستخدم (UI/Frontend) أو حتى طبقات العرض والقراءة (Projections) قبل اكتمال مرحلة البنية التحتية الخلفية (Backend Foundation) بشكل كامل وقابل للتشغيل الفعلي المتكامل (End-to-End).

**مراحل البناء والترقية المتعاقبة:**
1. **قاعدة PostgreSQL وحوكمة العزل المالي:** (Tenant Isolation + RLS + Core Data Integrity + Audit Base).
2. **محرك المحاسبة المركزي المغلق:** (Double-Entry Posting Engine + Immutable Ledger + Reversals).
3. **نطاق عمل فردي ومكتمل (Single Domain MVP):** تنفيذ نطاق عمل واحد فقط (مثل Inventory أو Sales) كحالة استخدام متكاملة بدون أي طبقات عرض أو واجهات مستخدم.
4. **طبقة الأحداث المعزولة:** (Outbox Pattern + Event Publishing) لتعمل من البداية للنهاية.
5. **طبقة الإسقاط والقراءة الأحادية:** (One Simple Read Projection Rule) لبناء نموذج قراءة مبسط للغاية.
6. **بوابة واجهة برمجة التطبيقات واستقرارها:** (API Gateway Stabilization).
7. **واجهة استخدام أولية محدودة للغاية:** (Frontend Minimal Flow) مخصصة فقط لنطاق العمل المكتمل.
8. **التوسع التدريجي في النطاقات الإضافية:** (Horizontal Domain Expansion) المخطط لها تتابعيًا بعد استقرار عصب النظام المالي.

### أ. طبقة قاعدة البيانات (PostgreSQL - Source of Truth)
* **الفلسفة والمسؤولية:** هي المستودع النهائي والكامل والوحيد للحقيقة التشغيلية والمالية. يقتصر دورها في البناء الأولي على تأمين البيانات، فرض تكامل القيود (Data Constraints)، ومراقبة الاتساق الذاتي للبيانات.
* **البنية التحتية للعزل (Multi-Tenancy Isolation):**
  * فرض العزل المادي المنطقي باستخدام **Row-Level Security (RLS)** لجميع الجداول عبر حقل `tenant_id` المعرف بصيغة UUID.
  * كل مستعلم أو عملية تتصل بقاعدة البيانات مجبرة على مواءمة الجلسة عبر ضبط متغير البيئة المحلي الحركي الخاص بالجلسة `SET LOCAL app.current_tenant_id = 'xxxx-xxxx-xxxx-xxxx'`.
* **تتبع الزمن الثنائي (Bitemporal Tracking Feature):**
  * تلتزم الطبقة بتسجيل فترتين زمنيتين منفصلتين للمعلومات الحساسة والمخازن: **Valid Time** (زمن سريان الواقعة الفعلي في بيئة الأعمال) و **Transaction Time** (زمن تسجيل الصفقة الحسي في النظام المحاسبي). يمنع هذا التلاعب بأرصدة الفترات المفتوحة والمغلقة بأثر رجعي.
* **ثبات البيانات المالية (Financial Immutability Constraint):**
  * يمنع منعًا باتًا إجراء عمليات `UPDATE` أو `DELETE` على أرصدة ودفاتر الأستاذ العام والقيد المزدوج. التعديل مسموح فقط وحصريًا عن طريق تسويات عكسية وإطاحة متعادلة وموقعة بقيد مقابل (Reversal entries) مرتبط برقم العملية الأصلي عبر خط هداية فريد `correlation_id`.
* **سياسة القفل والسباق المتزامن (Concurrency and Locking Policies):**
  * استخدام **Optimistic Locking** للمستندات التشغيلية والطلبات عبر رقم الإصدار `version`.
  * فرض **Pessimistic Locking** (`SELECT ... FOR UPDATE`) للعمليات الحساسة وعالية التردد التي تمس الأرصدة المخزنية والمالية اللحظية (مثل سحب الكميات والترحيل المحاسبي للبضائع لمنع حدوث Race Conditions والوصول للرصيد السالب خارج نطاق السداد المقبول).
  * مستوى عزل المعاملات المالي الافتراضي هو **Serializable** للترحيلات المحاسبية والقيود اليومية لضمان التسلسل التام، و **Read Committed** لباقي العمليات التشغيلية والتقارير.

* **ما هو مسموح:**
  1. القراءات والافرازات المرتبطة فقط بمالك البيانات المالك الفعلي في RLS.
  2. فرض النزاهة المرجعية عبر Foreign Keys والمفاتيح المدمجة.
  3. تنفيذ دوال الفلترة الملازمة لقاعدة البيانات لضمان عدم هروب البيانات بين فروع العميل.

* **ما هو ممنوع:**
  1. كتابة أي منطق أعمال (Business Rules) في شكل Triggers أو Stored Procedures معقدة تحسب أو تعدل قيم المبيعات أو الضرائب بصيغة حركية تفوق القيود المرجعية الصرفة.
  2. استدعاء خدمات خارجية (External API Calls) من داخل قاعدة البيانات.
  3. وجود حسابات مشتتة أو جداول وهمية لا تخضع لهيكل التحكم بالعزل (Tenant Control).

### ب. طبقة خدمات النطاق (Domain Services Layer)
* **الفلسفة والمسؤولية:** هي المالك الوحيد لمنطق الأعمال (Business Rules) والسياسات التفصيلية للشركات. كل نطاق عمل يعيش في جزيرة برمجية معزولة تمامًا (Modular Encapsulation).
* **قاعدة عزل النطاقات وتكامل المجاميع الجزرية (Domain Boundaries & Aggregates):**
  * يُمنع منعًا باتًا وجود علاقات مباشرة (Foreign Keys or Joins) عبر الجداول الخارجية المباشرة بين النطاقات المستقلة.
  * الاتصال بين النطاقات يتم حصريًا إما تزامنياً عبر واجهات برمجية موثقة وخاضعة للرقابة المعمارية (gRPC / Domain Interfaces) أو عبر أحداث التكامل اللاتزامنية (Integration Events).
* **إطار عمل وحدة الخدمة (Unit of Work Scope):**
  * تنفذ الطبقة نمط المعاملة الذرية (Atomic Unit of Work)، فكل طلب تغيير إما أن يُطبَّق بالكامل (Data Engine + Outbox Store) أو يُلغى بالكامل لضمان عدم تشوه اتساق المعلومات في لحظات السقوط التقني.

* **ما هو مسموح:**
  1. صياغة قواعد التحقق وضمان سلامة الكيانات (Domain Invariants Validation).
  2. تشكيل حالات الطلب وحوكمة دورات حياة المعاملات والسياسات الحركية للشركات.
  3. استدعاء محرك المحاسبة المركزي فقط عبر واجهته المعرفية لتسجيل الأثر المالي لعمليات النطاق.

* **ما هو ممنوع:**
  1. الوصول المباشر لقاعدة البيانات خارج مظلة المستودعات المصممة (Repositories Integration).
  2. إرسال أو نشر أي أحداث (Events) مباشرة إلى الخارج دون حفظها وتوثيقها داخل محطة الخروج المحلية (Transactional Outbox Storage) في نفس معاملة الترحيل المحاسبي.

---

### ج. طبقة بوابة التطبيق وجدران الفصل والتحقق (API Gateway / BFF Layer)
* **الفلسفة والمسؤولية:** نقطة الدخول الموحدة، وحارسة البوابة الحركية للنظام بالكامل. لا تعرف ولا تفهم تفاصيل قواعد الأعمال العميقة، بل تنظم حركة المرور وتضمن سلامة شكل وهويات المعاملات الواردة.
* **المهام الحيوية:**
  1. **التحقق من الهوية والصلاحيات:** تفكيك رموز التحقق (Authentication Tokens) والتحقق من سريان مفعول الجلسات.
  2. **تفويض الصلاحيات (Authorization Delegation):** التحقق الأولي الظاهري القائم على المسارات والوظائف المسندة للمستخدم.
  3. **تنظيم الحدود ومنع الإغراق (Throttling & Rate Limiting):** بناءً على الهوية والـ Tenant ID لمنع تضرر أداء العملاء الآخرين من سوء تخطيط الطلبات للعميل الواحد.
  4. **حقن تتبع العمليات (Correlation ID & Metadata Injection):** إنتاج وحقن رقم تتبع فريد عالمي غير قابل للتحوير يواكب كـ الوشم كافة المسارات البرمجية عبر جميع المكونات والبنى التحتية للمراقبة.
  5. **التحقق الهيكلي الصارم:** التدقيق على مدخلات المستخدمين ومقارنتها بمخططات الهياكل المعيارية (JSON Schema/Protobuf Validation) في الواجهة الأمامية للبوابة.

* **ما هو ممنوع:**
  1. اتخاذ قرارات مالية أو حساب قيم الفواتير أو تسويات الأرصدة.
  2. الاتصال مباشرة بمستودعات البيانات (PostgreSQL) أو تجاوز طبقة الخدمات الأساسية (Domain Layer).

---

### د. طبقة معالجة وبث الأحداث (Event Layer - Outbox & Kafka)
* **الفلسفة والمسؤولية:** هي النظام العصبي القائم على حفظ الأحداث وبثها وضمان سلامة التسليم اللاحق، وحرص الاتساق النهائي للبيانات (Eventual Consistency).
* **تصميم صندوق المعاملة المحلي (Transactional Outbox Pattern):**
  * لضمان عدم حدوث تشوهات ماليّة، يتم إدخال وحفظ حدث التكامل (Integration Event) في جدول مخصص `outbox_events` كصف منبثق من نفس المعاملة المحلية لقاعدة البيانات التي كتبت بيانات الأعمال (مثال: حفظ الفاتورة + حفظ حدث إصدار الفاتورة يتمان معًا في معاملة ذرية واحدة بـ PostgreSQL).
* **ضمانات التسليم والتحصين ضد التكرار (At-Least-Once Delivery & Idempotency):**
  * الضمان التشغيلي هو **At-Least-Once Delivery**. المستهلكون (Consumers) مصممون بعناية ليكونوا **Idempotent Consumers**.
  * يتم ذلك عبر فحص المعرف الفريد لكل حدث وتسجيله بجدول معالجة الصدمات المكررة `idempotency_audit_registry` محمي بحظر مفتاح فريد لمنع تشغل المعاملة ذاتها مرتين.
* **مرونة التقييم وإدارة الأخطاء (Retry Boundaries & DLQ):**
  * معالجة الأخطاء تتبع استراتيجية التراجع التدريجي الأسي والضوضاء المعوضة (Exponential Backoff with Jitter).
  * بعد استنفاذ أربع تكرارات تصاحبها دورة فشل، تُرسل الرسالة المنهكة تلقائياً لمستودع الأحداث المعطلة (Dead Letter Queue - DLQ) مع تنبيه تتبع تكتيكي فوري لفتح ملف تفتيش بشري للتحقيق وإيقاف السقوط المتتالي للمهام (Circuit Breaker Integration).

---

### هـ. طبقة القراءة الموزعة وإسقاط الحسابات (Projection Layer - CQRS Read Side)
* **الفلسفة والمسؤولية:** هي المسؤولة عن تحويل الأحداث التاريخية المتراكمة إلى نماذج قراءة سريعة، مستهدفة بشكل دقيق تسهيل الاستعلام البشري وبناء لوحات المتابعة الحركية الآنية (Real-time Dashboards) من خلال أنظمة مخصصة سريعة الاستجابة مثل **Firestore** كأداة إسقاط مرئي فقط لمستخدمي المتصفحات.
* **سياسة التزامن والاتساق الزمني (Eventual Consistency SLA):**
  * اتفاقية سداد الخدمة الصارمة للمزامنة بين قاعدة البيانات الحية (SSoT) وطبقة العرض الموزعة (Firestore Projections) هي **SLA < 2.0 Seconds** في أسوأ ظروف التحميل الاستثنائية للشبكات.
* **الحوكمة عند التأرجح الاستعلامي (Read Lag Handling):**
  * في حال انتظار ترحيل مالي لم يتكامل حركيًا بعد مع Firestore، ينفذ النظام حوكمة واجهة المستخدم حيث تُمنح الواجهات مؤقتًا نظرة ترقبية استباقية محتسبة محليًا بالمتصفح مع علامة تتبع تهمس للمستخدم بـ "جاري اعتماد المعاملة بالدفاتر والمزامنة..." لمحو فجوة التباطؤ اللحظي (Optimistic UI Updates with Integrity Flags).
* **إعادة بناء النماذج (Projection Rebuild Strategy):**
  * عند إجراء صيانة أو إعادة هيكلة كاملة لنطاق التقارير والقراءات، يمتلك النظام آلية لإعادة تشغيل شريط الأحداث التاريخية منذ نقطة الصفر أو طوق زمني محدد ليعاد ملء نماذج العرض والصور اللحظية بالكامل دون المساس بسلامة واستقرار السجلات المركزية في PostgreSQL.

### و. طبقة العرض والواجهات الأمامية (Frontend Layer)
* **الفلسفة والمسؤولية:** واجهة خرساء (Dumb Presentation UI) لا تخزن قيم الهوية أو الصلاحيات بشكل مستقل ولا تحل منطق الأعمال.
* **المحرمات الصارمة:**
  * يمنع منعًا باتًا حساب نسب الضرائب، احتساب أرصدة حسابات الدفتر اليومي، التحقق الذاتي من مستويات الصلاحيات للأزرار خارج نطاق الحظر البصري التوجيهي، ويحرم الاتصال المباشر بأي شكل بقواعد البيانات أو تجاوز خدمات الموثوقية بالبوابة (BFF).

## 2. مصفوفة النطاقات المعزولة (Bounded Contexts Architecture)

لكل نطاق عمل قيود صارمة وحدود بيانات محكمة لضمان عدم حدوث تلوث معنوي أو تشابك برمجي بين الوحدات.

```
       ┌────────────────────────────────────────────────────────┐
       │             Master Data Management (MDM)               │
       │  (Owns Customers, Vendors, Items, Taxes, UOM Masters)  │
       └────────────────────────────────────────────────────────┘
          │ (Read-Only Copy / Event Sync)
          ├───────────────────────────────────┐
          ▼                                   ▼
┌──────────────────┐               ┌──────────────────┐
│ Inventory Domain │               │   Sales Domain   │
│ (Owns Stock Vals)│               │  (Owns Invoicing)│
└──────────────────┘               └──────────────────┘
          │                                   │
          │ (Post Financial Journal Entry)    │
          └─────────────────┬─────────────────┘
                            ▼
       ┌────────────────────────────────────────────────────────┐
       │               Accounting Core Domain                   │
       │  (Only Domain Permitted to Commit Ledger Transactions) │
       └────────────────────────────────────────────────────────┘
```

### أ. نطاق المحاسبة المركزي والمالية (Accounting Domain - CORE - Aggregate: JournalEntry)
* **مسؤوليات النطاق:** الإشراف المطلق والحصري على كافة الأرصدة المالية والحركات الدفترية ومستودعات الأستاذ العام وتطبيق الحوكمة المحاسبية.
* **الأرصدة ومجموعات البيانات المملوكة:**
  الدليل المحاسبي (Chart of Accounts)، القيود اليومية (Journal Entries)، بنود القيود المحمية (Journal Lines)، الفترات المالية وحوكمة الصلاحيات الفترية (Fiscal Periods).
* **المستندات والمعاملات المحظورة:**
  يُحظر على هذا النطاق بشكل تام امتلاك معلومات دقيقة تتعلق بسير خطوط الإنتاج التصنيعية، تفريغ الشحنات وتنسيق الأرفف اللوجستية، تفاصيل الإجازات الممنوحة للموظفين، أو بيانات العملاء الباردة في نظام CRM.
* **بروتوكولات التفاعل (Integration Contracts):**
  يستهلك الأحداث المالية الواردة من النطاقات الأخرى بشكل غير متزامن، ويقدم واجهات برمجية صارمة ومغلقة ومؤمنة بالكامل (gRPC APIs with Transaction Wrappers) لترحيل وتوليد القيود المالية الذرية من النطاقات المسندة.
* **حقوق التصرف المالي وسلطته الصارمة:**
  **هو الكيان الوحيد والنهائي داخل المطبخ المعماري بالكامل الذي يمتلك الحق الموثق للكتابة أو لمس الدفاتر والقيود المالية.** لا توجد أي قناة مباشرة لأي نطاق آخر لإحداث تفاعل مالي أو تدوير مالي إلا عبره وبموافقته الصارمة.

### ب. نطاق المخازن والمستودعات (Inventory Domain - Aggregate: WarehouseStock)
* **مسؤوليات النطاق:** مراقبة الأبعاد المادية والكمية وحركة دخول وتوطين وصرف البضائع والمنتجات.
* **الأرصدة ومجموعات البيانات المملوكة:**
  جداول المخزون والكميات المتاحة والمحجوزة والملغاة، مواقع التخزين داخل ثنايا المستودع، وثائق حركات المخزون المعيارية (Stock Movement Ledger)، حركات الإتلاف والجرد الفعلي والتسويات الكمية.
* **المستندات والمعاملات المحظورة:**
  أرصدة العملاء المالية والائتمانية الدقيقة، السجلات الجنائية والصحية أو كشوف الأجور للعاملين، وعقود مبيعات الشركات وأسعار صفقات الشراء المفتوحة.
* **بروتوكولات التفاعل (Integration Contracts):**
  يرسل بعد تحديث الأرصدة أحداث حيوية مثل `InventoryReserved` أو `StockValueRevaluated` لترجمتها محاسبياً وتسجيل فروق التقييم لمدمني الحسابات في النطاق الرئيسي.

### ج. نطاق المشتريات والتوريد (Purchasing Domain - Aggregate: PurchaseOrder)
* **مسؤوليات النطاق:** إدارة خطوط الطلب المالي وسلاسل التوريد ومعاملات إبرام الاتفاقيات مع الموردين.
* **الأرصدة ومجموعات البيانات المملوكة:**
  أوامر الشراء (POs)، طلبات العروض والتسعير الفنية، تصنيف المشتريات وبنود التسليم وسعر الصرف المتعاقد عليه لشراء الأصول والمواد التشغيلية.
* **المستندات والمعاملات المحظورة:**
  لا يعرف تفاصيل حساب البنك المالي المفتوح للشركات، مستندات الرواتب المحاسبية، أو طلبات الشحن الخارجي المحجوزة للمبيعات الصادرة.
* **بروتوكولات التفاعل (Integration Contracts):**
  عند ترحيل ومعاينة وثيقة الاستلام البضاعي المشتراة، يُطلق النطاق حدث `GoodsReceived` الذي يتابعه نطاق المخازن لرفع كمياته، ونطاق المحاسبة لإثبات التزام الدين المالي القائم على المشتريات.

### د. نطاق المبيعات والعملاء (Sales Domain - Aggregate: SalesInvoice)
* **مسؤوليات النطاق:** تنظيم هيكل الطلبيات الصادرة، حوكمة تسعير المبيعات التجارية، وإصدار الفواتير الرسمية المعترف بها.
* **الأرصدة ومجموعات البيانات المملوكة:**
  أوامر المبيعات (Sales Orders)، جداول عقود البيع، فواتير المبيعات الرسمية والمسودات، وتطبيقات خطط التخفيضات والعروض المؤقتة.
* **المستندات والمعاملات المحظورة:**
  لا يحق له معرفة أو التدخل في تكاليف الإنتاج التصنيعية، خطط الرواتب والعمل الإضافي، أو تكاليف الشراء والشحن والعمولات الداخلية المفصلة لشركات النقل اللوجستي.
* **بروتوكولات التفاعل (Integration Contracts):**
  يقدم عقود المبيعات لجميع النطاقات عبر أحداث التكامل والربط مثل `InvoiceIssued` و `SalesOrderApproved` التي تعبر صياغتها للمحاسبة والمخازن لتحريك الأجنحة التشغيلية المصاحبة فوراً.

### هـ. نطاق إدارة الموارد البشرية والرواتب (HR Domain - Aggregate: EmployeePayroll)
* **مسؤوليات النطاق:** تدبير ملفات الموظفين، السير الوظيفية، ساعات الحضور والمغادرة، تسيير الأجور الشهرية واحتساب التأمينات الاجتماعية والضرائب العمالية.
* **الأرصدة ومجموعات البيانات المملوكة:**
  سجلات الموظفين، عقود العمل والشؤون الوظيفية، كشوف الأجور الشهرية المعقدة (Payroll Registers)، حركات الحضور والغياب، ومذكرات طلبات السلف المالية.
* **المستندات والمعاملات المحظورة:**
  أرصدة وكميات المستودعات، حسابات الموردين والشحن، أو أسعار تذاكر مبيعات نقاط البيع.
* **بروتوكولات التفاعل (Integration Contracts):**
  عند غلق واعتماد مسودة كشف الأجور الحركية، يطلق حدث `PayrollApproved` ليمكِّن نطاق المحاسبة من تشكيل قيد الاستحقاق العريض للأجور والمنافع وتصديره لمركز الحسابات.

### و. نطاق إدارة علاقات العملاء (CRM Domain - Aggregate: CustomerOpportunity)
* **مسؤوليات النطاق:** استيعاب وتسيير العلاقات الأولية مع العملاء من فرص ومفاوضات ومتابعة الاتصالات قبل الترسية التجارية وفواتير البيع.

### ز. نطاق نقاط البيع (POS Domain - Aggregate: POSSession)
* **مسؤوليات النطاق:** تسوية ومتابعة عمليات البيع المباشر والكاش في الفروع والمحطات الحية والطلبيات السريعة.

### ح. نطاق التصنيع والإنتاج (Manufacturing Domain - Aggregate: ProductionOrder)
* **مسؤوليات النطاق:** تسيير خطوط التشغيل وتحويل المكونات الأولية إلى سلع نهائية وتوزيع وحساب أعباء التصنيع.

### ط. نطاق إدارة المشاريع (Projects Domain - Aggregate: EnterpriseProject)
* **مسؤوليات النطاق:** حوكمة الأعمال الهندسية والزمنية والمهام والميزانيات المستقلة للمشروعات التجارية والإنشائية.

### ي. نطاق إدارة البيانات المرجعية الحاكمة (Master Data Management - MDM - Aggregate: MDMRegistry)
* **المالك الحصري الخالي من التشويه:** يمتلك السيادة الكاملة ولا يقبل القسمة أو التعديل من النطاقات الأخرى على:
  * سجل الموردين الاستراتيجي (Vendor Master).
  * سجل العملاء الموحد (Customer Master).
  * سجل المواد والمنتجات ومواصفاتها (Item Master).
  * سجل وحدات القياس (UOM Master) وعوامل التحويل الرياضي بينها.
  * سجل الضرائب والأوعية الخاضعة للتشريع الحكومي (Tax Master).
* **القوانين الصارمة للـ MDM:**
  لا يحق لأي وحدة برمجية أو نطاق تشغيلي إنشاء سجل عميل أو مورد أو وحدة هندسية أو صنف مخزني مخصص أو تعديله داخليًا في جداوله المستجوب عزلها. التعديل يتم مركزيًا في هذا النطاق الحارس، وتستهلك باقي الوحدات هذه السجلات الموثقة بصيغة **Read-Only Contracts** وتتزامن مع تحديثاتها عبر بث الأحداث الحركية الموقعة من النطاق المرجعي.

### ك. نطاق حوكمة محرك موافقات مسارات العمل (Workflow Engine Domain)
* **المسؤوليات الحصرية:** إدارة وتخزين سلاسل الموافقات الحركية متعددة المستويات وقواعد التصعيد الوظيفي (Escalation Rules) عند تجاوز المهلة، وتوثيق التفويضات المؤقتة للمناصب المعزولة.

### ل. نطاق إدارة وأرشفة المستندات والوثائق والحيازة (Document Management Domain)
* **المسؤوليات الحصرية:** تشفير وحفظ المرفقات المادية وصور الفواتير والتعاقدات الرسمية، وتحليلها آليًا عبر أنابيب القراءة والتصنيف، وضمان ثبات وحصانة مستندات المراجعة والقوانين ضد إهدار التخزين.

### م. نطاق إعداد التقارير والتحليلات العليا (Reporting & Analytics Domain)
* **المسؤوليات الحصرية:** بناء وتحريك التقارير المالية والتشغيلية المعقدة (ميزانيات عمومية، تدفقات نقدية، كشوف أرباح مجمعة) وتنسيق حساب مؤشرات الأداء العليا للقيادات من جداول الإسقاط المعزول في بيئة آمنة تمنع المساس بنبض PostgreSQL المحاسبي اليومي.

## 3. محرك المحاسبة المركزي والقيود المزدوجة (Accounting Engine Design)

الرقابة الدقيقة على الميثاق المالي تقتضي تصميم هيكل الحسابات كصندوق مغلق حامي للقيم الاستحقاقية.

```
       ┌────────────────────────────────────────────────────────┐
       │                Accounting Engine API                   │
       │     (gRPC Endpoints, Input & Strict Validation)         │
       └────────────────────────────────────────────────────────┘
                                   │
                                   ▼
       ┌────────────────────────────────────────────────────────┐
       │             Posting Pipeline / Validations             │
       │     1. Fiscal Period Closed Check?                     │
       │     2. Debit Sum == Credit Sum?                        │
       │     3. Tenant ID Integrity Verified?                   │
       └────────────────────────────────────────────────────────┘
                                   │
         (Balanced & Locked) ──────┼────── (Failed/Violated)
         ▼                               ▼
┌────────────────────────────────┐ ┌────────────────────────────┐
│      Commit Journal Entry      │ │     Throw Error Reject     │
│  (Database Immutability, GL)   │ │  (Audit Violation Logged)  │
└────────────────────────────────┘ └────────────────────────────┘
```

### أ. الركائز الأساسية لمحرك القيد المزدوج:
1. **توازن كافتي الميزان (Balanced Journals Constraint):**
   كل قيد محاسبي يلتزم بالقانون المعياري الأصيل: مجموع قيم الطرف المدين (Debit) يساوي تمامًا وبصيغة مطلقة مجموع قيم الطرف الدائن (Credit) لنفس معاملة القيد `JournalEntry` على مستوى العملة التشغيلية الموحدة.
2. **بروتوكول حظر التحديث والحذف المباشر (Immutable Ledger Contract):**
   * بمجرد تحول القيد المحاسبي من حالة مسودة إلى مرتبة "مُرّحل" (Posted) ويتم اعتماده وكتابة المعاملة في PostgreSQL، يغلق السطر برمجياً ومادياً.
   * لا يمكن إطلاقاً وحتّى لأعلى منصب إداري في النظام استخدام تعبيرات `UPDATE` أو `DELETE` لاستبدال الأرقام أو تعديل طبيعة السطور المالية.
3. **التصميم العكسي والاسترجاع بالتسويات (Reversal-Only Correction Policy):**
   * عند الوقوع في خطأ مالي أو رغبة المستخدم في إلغاء وتعديل حركة ما، يتم صياغة وإطلاق معالجة عكسية رسمية (Reversal Journal Entry) تحمل أرقامًا سالبة في الأماكن المقابلة أو تبديل الحسابات المدينة والدائنة بقيم موازية تماماً للأصلية لإلغاء المفعول الحسابي، مع ربط دائم بالمعرف الفريد للأصل المحذوف حوكمته في الجدول تحت الحقل `reversed_journal_entry_id` والرقم التسلسلي `correlation_id`.
4. **حوكمة شروط الإغلاق وتوفر الفترات المالية (Fiscal Period Enforcement):**
   يكتشف المحرك تلقائيًا حالة الفترة المالية الحاكمة لزمن القيد وفي حال تسجيل تاريخ يقع ضمن فترة تحت حالة "مغلقة مؤقتًا" (Soft-Locked) أو "مغلقة كليًا" (Hard-Locked)، يجهض المحرك المعاملة فورًا ويرتب قيد مخالفة بروتوكولية في سجل الأمان.

---

### ب. تفاصيل دورات الحياة والمستويات التشغيلية:

#### دورة حياة القيد اليومي (Journal Entry Lifecycle):
1. **Draft (مسودة):** يحق للنطاق المنشئ التحديث والتنزيل واختبار مطابقة كفة الأرقام والتجربة الفنية.
2. **Validated (موثق ومفحوص):** يجتاز القيد فحص التوازن الحسابي الذاتي للسطور والتحقق من حسابات الدليل المحاسبي وحالة الفترات وتوفر الأمان للـ Multi-Tenant.
3. **Posted (مُرّحل):** يُلتزم بكتابة القيد في السجل المالي والأثر التراكمي لدفتر الأستاذ العام وتاريخ الزمن المتداخل وحظر لمس السطر نهائيًا وإطلاق الأحداث المعاونة للجهات الخارجية.
4. **Locked (مغلق نهائيًا):** إثر انقضاء غيار دورة الفترة الجارية وتثبيت المراجعة المالية، يُقفل نطاق الفترات برمتها لتجميد حالة السجلات بالكامل.

```
       ┌──────────┐
       │   Draft   │
       └─────┬─────┘
             │ (Submit & Verify Balancing/Period)
             ▼
       ┌───────────┐
       │ Validated │
       └─────┬─────┘
             │ (Post Command - Run transactional SSoT commit)
             ▼
       ┌───────────┐
       │  Posted   │  ──► (Immutable Record, No Update, No Delete)
       └─────┬─────┘
             │ (Fiscal Period Archival / Finalizing Ledger)
             ▼
       ┌───────────┐
       │  Locked   │
       └───────────┘
```

#### مسار تيسير عمليات المحاكاة والترحيل المحاسبي (Posting Pipeline):
* تتم معاملة البورصة المحاسبية للتأمين والترحيل كعملية واحدة متراصة تحت سقف معاملة PostgreSQL معزولة ومحمية ومتحضرة ومغلقة محليًا لمنع حدوث تناقض للأرقام أو تشوهات جزئية للحركات تحت فجوة سقطات الشبكة.

#### سياسة ومعالجة العملات الأجنبية وأسعار السداد (Multi-Currency & FX Rate Policy):
* يتم تخزين كل سطر محاسبي بثلاث طبقات متوازية من الأرقام لتقليل فجوات التقريب الرياضية:
  1. القيمة بعملة المعاملة الحركية (Source Amount & Currency).
  2. القيمة بعملة التقرير المحلية الأساسية للشركة (Base System Amount) بسعر الصرف في وقت تسجيل الحركة المعتمد من MDM.
  3. القيمة بعملة التقارير المجمعة للشركة القابضة (Consolidated Amount).
* معالجة أرباح وخسائر فروق أسعار الصرف غير المحققة (Unrealized FX Gain/Loss) تتم آلياً في نهاية كل فترة مالية عبر محرك التسويات الآلي.

#### حدود الحوكمة الضريبية وفصل القواعد (Tax Isolation Boundary):
* لا تشارك النطاقات التشغيلية (مثل المبيعات) في عمليات صياغة قوانين احتساب الضرائب الحكومية أو معالجة اقتطاع ضريبة القيمة المضافة من المنبع.
* يمرر طلب النطاق المحجوز مع مبالغه الخام ويتولى محرك المحاسبة وعبر مصلحة الضرائب بالـ (Tax Master) احتساب وتفتيت الخيوط والقيام بالتوجيه التفصيلي للحساب الدائن للضرائب دون ترك فرصة للخطأ البشري أو تسريب قواعد الضرائب للواجهات والبرمجيات التشغيلية الصغرى.

## 4. استراتيجية مسارات العمل الممتدة عبر النطاقات وسجلات المواءمة (Saga Pattern)

لتنسيق العمليات والأنشطة التي تمتد لتطرق عدة نطاقات برمجية دون استخدام معاملات قاعدة بيانات متباعدة ومشتتة (Distributed Transaction)، نلتزم بنمط **Saga Orchestration** حارس ومراقب للمسارات المتقلبة.

```
[Sales Domain] ──► (Create SO) ──► [Saga Coordinator] ───────┐
                                         │                    │ (Reserve Inventory)
                                         │                    ▼
                                         │            [Inventory Domain]
                                         │            (Success or Rollback)
                                         │                    │
                                         │◄───────────────────┘
                                         │
                                         │ (Post Financial Invoice)
                                         ▼
                             [Accounting Core Engine]
```

### أ. حوكمة وإرساء نمط التوزيع السحابي والتحوير الفني المعوض:
1. **أخذ القرار بالمركزية والتنسيق (Saga Orchestration Policy):**
   * يُفضَّل دائمًا استخدام محرك المنسق المركزي (**Orchestrator-Based Saga**) للعمليات الحساسة والمعقدة متعددة الحواضر (مثل سداد الطلبات وتوفير المخزون وإصدار فواتير الديون) لإشرافه الكامل وتخزينه المستقل للـ (Saga State Tracker) وسهولة تقفي أثر سير المعاملة بدقة ومقاومة الضياع والتهويم المعرّف.
2. **قواعد الإطاحة وتسويات التراجع المحاسبي (Compensating Transactions):**
   * بما أن المعاملات البينية موجهة بشكل غير تزامني وتلتزم بالاتساق النهائي، فإن حدوث فشل في خطوة متأخرة بالمساق يفرض إيقاظ معاملة تعويضية جراحية (Compensating Transactions) تعود بالخطى المعكوسة لإلغاء ما تم مسبقا لضمان اتساق النظام الإجمالي.

---

### ب. أمثلة لمسارات سلاسل الحوكمة (E2E Sagas & Recovery Steps):

#### 1. دورة حياة طلب البيع (Sales Order Cycle):
* **خطوات المسار الأساسي التتابعي:**
  1. طلب البيع ينشأ في نطاق المبيعات (`SalesOrderCreated`).
  2. يتم حجز الكميات المخزنية المطلوبة مؤقتًا في نطاق المخازن بنجاح (`InventoryReserved`).
  3. يتم تفجير واحتساب الفاتورة وترحيل الأثر المالي في حسابات المحاسبة العمومية للعميل (`InvoicePosted`).
  4. استيعاب الترحيل المحاسبي لنبض الدفع المالي وسداد الفاتورة المعتمدة بكفالة البنوك (`PaymentCaptured`).
* **استراتيجية التعويض في لحظات السقوط والفشل:**
  * إذا سقطت المحاسبة أو عجزت المعاملة عن قيد الفاتورة التراكمية، يُوجَّه المنشط لإطلاق معاملة تعويضية فورية تأمر نطاق المخازن بـ "فك الحجز الكمي" للمواد المعزولة وتحديث الجدول لفك العزلة وعودة كميات وصناعة البضاعة للرصيد السائب الحر (`CancelStockReservationCommand`).

#### 2. دورة سلاسل التوريد والشراء (Procurement Workflow):
* **خطوات المسار الأساسي التتابعي:**
  1. صياغة مذكرة الحاجة في المشتريات وتعميم الموافقة الإدارية عليها (`PurchaseRequisitionApproved`).
  2. ولادة وثيقة شراء معتمدة للمورد بسعر صرف وتكلفة موثقة وتوطيد أمر التوريد (`PurchaseOrderPosted`).
  3. تفريغ واستلام شحنات المواد المخزنية وقيدها في السجلات المستودعية وفحص سلامتها (`GoodsReceived`).
  4. ترحيل حساب الدائنين والخصومات وإثبات نشوء المسؤولية والالتزام المالي بالدفاتر (`A/P Liability Credited`).
* **استراتيجية التعويض في لحظات السقوط والفشل:**
  * إذا تم رفض الجودة أثناء التفريغ وفحص المواد وتوقفت المعاملة، يصدر المحرك تراجعاً ماليًا لإلغاء الالتزامات وقيد استحقاق المشتريات المعطوبة مع ترحيل إشعار دائن بديل لحساب النطاق المحاسبي.

## 5. ميثاق معالجة وتثبيت الأحداث الموزعة (Event-Driven Specification)

```
                            Event Publishing Workflow
                            
┌─────────────────────────┐     ┌─────────────────────────┐     ┌─────────────────────────┐
│  State Change Committed │ ──► │ Write Outbox Event Reg  │ ──► │  Outbox Relayer Worker  │
│  (Database Domain Tx)   │     │ (Same Transaction SSoT) │     │ (Asynchronously Reads)  │
└─────────────────────────┘     └─────────────────────────┘     └─────────────────────────┘
                                                                             │
                                                                             ▼
                                                                ┌─────────────────────────┐
                                                                │  Publish to Event Bus  │
                                                                │   (Kafka Topic Stream)  │
                                                                └─────────────────────────┘
```

1. **انقسام وتصميم الأحداث بحدود الصلاحية (Domain vs Integration Events):**
   * **Domain Events:** أحداث مجهرية تنشأ وتموت بذاتها داخل النطاق المغلق المعين لربط كياناته (مثل: `JournalLineValidated`). لا تخرج خارج الحدود لتلوث النطاقات المجاورة.
   * **Integration Events:** أحداث شمولية عامة خاضعة لسجلات العقود ومُعرَّفة بعناية للنشر عبر بيئة النظام لمتابعة الأثر المتباعد (مثل: `FiscalPeriodLocked`).
2. **تطهير واجهات السير وضمان البث الآمن (Outbox Engine Publishing):**
   * تلتزم كافة النطاقات بوجود محرك متتبع محلي يدعى (**Outbox Relayer Worker**) يقرأ سجل الأحداث المكتوبة لجدول المخرجات المحلي داخل PostgreSQL والذي تعهدت معاملات الأعمال بحفظه، ومن ثم نقلها وإيداعها على الخراطيم السحابية لـ (Kafka Topics) بشكل يضمن الفصل التام بين زمن المعاملة وسير اتصالات الشبكة والاتصال الخارجي.
3. **حوكمة فحص ومنع مأزق تكرار الأحداث (Idempotency and Order Guarantees):**
   * تُبث المهام والرسائل داخل قنوات وأقسام فرعية (Partitions) معتمدةً على مفتاح تنظيمي فرعي مثل `tenant_id:company_id` لضمان أن الأحداث لذات الكيان المعزول معالجتها تجري وراء بعضها بالتسلسل الزمني الصارم للتوليد دون تشتت برتب المعالجة في المستودعات البعيدة.

---

## 6. استراتيجية وبنيان العزل السحابي والآمان الشامل (Multi-Tenancy Core Architecture)

إن بقاء النظام واستحقاقه للحصانة يعتمد على قوة وموثوقية الفصل والتحصين المفروض تقنيًا وماديًا بين الكيانات والشركات المستضافة بقواعد البيانات المشتركة.

```
       ┌────────────────────────────────────────────────────────┐
       │                 Tenant Access Request                  │
       │           (Tenant ID & Authentication Checked)         │
       └────────────────────────────────────────────────────────┘
                                   │
                                   ▼
       ┌────────────────────────────────────────────────────────┐
       │              PostgreSQL Connection Pool                │
       │        (Inject Tenant ID in Session Context)           │
       └────────────────────────────────────────────────────────┘
                                   │
                                   ▼
       ┌────────────────────────────────────────────────────────┐
       │                Row-Level Security (RLS)                │
       │       SELECT * FROM journals WHERE tenant_id = RLS     │
       └────────────────────────────────────────────────────────┘
                                   │
                   ┌───────────────┴───────────────┐
                   ▼                               ▼
       ┌───────────────────────┐       ┌───────────────────────┐
       │   Tenant A Data Only  │       │   Tenant B Data Only  │
       │   (Secure & Isolated) │       │   (Secure & Isolated) │
       └───────────────────────┘       └───────────────────────┘
```

### أ. استراتيجيات عزل وحوكمة البيانات (Shared Database / Shared Schema Approach):
* نحن نعتمد وبما يخدم قنوات خفض التكاليف وتسريع صيانة البنى نظام الهيكل المشترك المعزول برمجيًا وماديًا (**Shared DB with Shared Schema**) محصنًا بأعتى درجات الحماية على السطور الجداول في المحرك الخادم عبر RLS.
* يحرم العبور والاستعلام بدون المرور الإجباري على مرشح العزل `tenant_id` داخل شق قاعدة البيانات.

### ب. قواعد العزل وقفل الشركاء والشركات التابعة (Isolation Hierarchy Rules):
1. **Tenant Isolation:** عزل كلي مطلق يمثل الحدود السحابية للعميل المستقل (لا تواصل، لا شراكة، لا تبادل أرصدة عبر الحدود إلا بفرد خادوم متداخل من طبقة ACL مصادق عليه).
2. **Company Isolation:** الشركات التابعة لنفس العميل السحابي (Tenant) كأجنحة قانونية مستقلة تماما بتقرير الميزانيات العمومية والدليل المحاسبي المنعزل.
3. **Branch & Warehouse Isolation:** فروع لوجستية ومواقع تخزين للأوعية والمستندات الحية كأبعاد فرعية وحركية لتنظيم حركة الأيدي العاملة داخل شريكة الجناح، مع حظر تداخل رؤية المستندات إلا لمستحقي رتب الوصول.

---

### ج. إستراتيجية توطيد الحفظ وأرشفة البيانات التراكمية (Data Lifecycle & Archiving Strategy):
* **Partitioning Strategy:** تُقسم حركات الجداول الكبرى لدفاتر الأستاذ العام والمعاملات استنادًا للتقويم السنوي الهجين والربط المعرف `tenant_id` والزمن `year_month` لضمان تسريع المؤشرات وقصر الاستعلام في PostgreSQL على أسطوانة الذاكرة المعروضة للتسيير الفعلي.
* **Cold Storage Policy:** بعد انقضاء سنة مالية كاملة وجعل رتبة العام المالي "مغلق نهائياً"، تُشحن وتُهجر السجلات الخاملة غير الحركة إلى مستودعات باردة مع الاحتفاظ الدائم والفوري بأرصدة الافتتاح لبقية السنوات والفرعيات دون التدمير الكلي للسلوك المالي للتاريخ.
* **المبادئ القانونية (Legal Hold & HIPAA/SOX Alignment):**
  يمنع تماما استخدام خواص الحذف للتثبيت القانوني المالي والسماح لملفات التحقيق المراجعة (Forensic Audit Requests) بالعودة لتفاصيل القيد في أي لحظة. السجلات تخضع لعمليات الحيازة وحظر التدمير (Legal Hold Option) بنمط تشفير موثق.

## 7. هيكل وتصميم نظام إرساء الصلاحيات المزدوج (Authorization RBAC & ABAC Hybrid)

لا ثقة في كود الواجهة الأمامية مطلقاً. يتم معالجة الرخص والصلاحيات استناداً لهجين متقدم من التحكم بالوظيفة والتحقق من السياق والكيانات الصارمة.

```
       ┌────────────────────────────────────────────────────────┐
       │               BFF Layer Authorization Check            │
       │           Roles (RBAC) + Attributes (ABAC)             │
       └────────────────────────────────────────────────────────┘
                                   │
          ┌────────────────────────┼────────────────────────┐
          │ (Validated Parameters) │                        │ (Violated Attributes)
          ▼                        ▼                        ▼
┌───────────────────┐    ┌──────────────────┐    ┌──────────────────┐
│     Tenant ID     │    │   Branch Scope   │    │ Reject Execution │
│  (Verify Access)  │    │  (Within Bounds) │    │  (Unauth Event)  │
└───────────────────┘    └──────────────────┘    └──────────────────┘
          │                        │
          └───────────┬────────────┘
                      ▼
       ┌────────────────────────────────────────────────────────┐
       │             Database Level RLS Filter App             │
       │            (Execution Allowed & Checked)               │
       └────────────────────────────────────────────────────────┘
```

1. **الوصول على أساس الدور (Role-Based Access Control - RBAC):**
   * إسناد رخص مهام وظيفية مغلقة ومعبأة مسبقاً (مثل: `Accountant` - محاسب، `WarehouseManager` - مدير مخزن، `SalesRepresentative` - مندوب مبيعات...).
2. **التحكم على أساس الخصائص الدلالية والسياقية (Attribute-Based Access Control - ABAC):**
   * التحقق المتقدم والديناميكي المصاحب للطلب: هل يملك المحاسب حق إقرار فواتير مبيعات تفوق قيمتها الحد الدلالي الممنوح له في نطاق حركتِه ($10,000 USD)؟ هل يقع الفرع المختار ضمن حيز مستخدَم هذا الموظف؟ هل الفترة المالية حالتها "تسمح بالقيد الحركي"؟
3. **ميثاق فض النزاعات وتضارب المصالح المالي (Separation of Duties - SoD Rules):**
   * **القاعدة الذهبية للنزاهة المالية:** يمنع النظام تقنياً وعبر فحوص صارمة في محرك الحسابات تداخل المهام المانعة لشفافية الحسابات؛ فمثلاً، يُمَنع برمجياً الموظف المحاسب الذي قام بصياغة وإنشاء مسودة القيد اليومي (`JournalDraftCreator`) من التوقيع بالاعتماد والترحيل المالي لذات القيد (`JournalEntryPoster`) لضمان غياب المؤامرات الحركية والتحصيل الاحتيالي.

## 8. طبقة فك التعارض مع الأنظمة والجهات الخارجية (Anti-Corruption Layer - ACL)

يحرم تسريب بنى نماذج البيانات للجهات والشركاء الخارجيين إلى نسيج النطاقات المغلقة في النظام ومستودعات البيانات المعزولة.

```
┌──────────────────────────────────────┐  ┌─────────────────────────────┐
│          External Bank API           │  │      Local Tax System       │
└──────────────────┬───────────────────┘  └──────────────┬──────────────┘
                   │                                     │
                   ▼                                     ▼
┌───────────────────────────────────────────────────────────────────────┐
│                    Anti-Corruption Layer (ACL)                        │
│                (Mapping, Translation, Circuit Breakers)               │
└──────────────────────────────────┬────────────────────────────────────┘
                                   │ (Standard Safe Domain Event)
                                   ▼
┌───────────────────────────────────────────────────────────────────────┐
│                        Core Monolith Gateway                          │
│                  (Protected Core Domain Integrity)                    │
└───────────────────────────────────────────────────────────────────────┘
```

* **مهمة طبقة فك التعرض وعزل التغيير (ACL Task Specification):**
  تعمل كجسر ترجمة متين ومستقل يتولى تحوير وسد الفجوات بين هياكل البيانات المتموجة الوافدة من البنوك، بوابات السداد الخارجي، النظم الضريبية في الدول المستهدفة بالإيصالات الرسمية الإلكترونية، وتحويلها لخرائط بيانات مع معاييرنا ونمذجة الأحداث الآمنة لدينا، وحماية المعالجة اللحظية من انهيار تلك الكيانات عبر حارس القطع التدريجي السريع (**Circuit Breaker**).

## 9. حوكمة فترات التشغيل والتقاويم المالية (Financial Period Management)

المخزن المالي مستقر بقوة التقاويم والتحكم بأبواب الغلق المحاسبي.

1. **تعدد التقاويم المالية والأنشطة المقارنة (Multi-Fiscal Calendars):**
   دعم بناء تقويم مالي متموج ومرن لكل شركة تابعة بما يطابق تشريع الدولة الجغرافي المعمول به دون الإخلال بالمزامنة بين الشركات التابعة.
2. **الغلق المرن والصلب للفترات الدفترية المحكمة:**
   * **Soft Lock (إغلاق ناعم):** تجميد مؤقت لعمليات الإجراء والكتابة الجديدة تمهيدًا لأعمال التدقيق والمراجعة، مع السماح بنشر التقارير واستطلاع البيانات للمدققين.
   * **Hard Lock (إغلاق حديدي):** غلق أبواب الدفاتر بصورة لا تسمح بأي محاولة إدخال، وقطع الطريق على إمكانية التسجيل نهائياً وحساب ميزان المراجعة والأرصدة الختامية للترحيل.
3. **حوكمة فتح الفترات المالية وتصاريح الاستثناء المتقاطعة:**
   تعد عملية إعادة فتح عام مالي مغلق أو فترة مالية من المحرمات الكبرى، ولا تمنح رخصتها الحركية إلا عبر إذن من المدير المالي للشركة مشروطةً بتوقيف العمليات الجارية وقيد مذكرات تبيين المخاطر وإرسال تنبيه جنائي وفوري لنظام التدقيق.

## 10. الرقابة والملاحظة والتدقيق الجنائي للأمان (Observability & Forensic Audit)

كل نبضة برمجية تثير تتبعًا، وكل لمسة ماليّة تشكل صفاً باقياً للتفتيش.

```
                                  Telemetry & Auditing Pipeline
                                  
┌───────────────────────────────────────────────────────────────────────┐
│     Incoming API Request with Correlation ID (BFF Gateway)            │
└──────────────────────────────────┬────────────────────────────────────┘
                                   │
                                   ▼
┌───────────────────────────────────────────────────────────────────────┐
│                     Active Execution Flow logs                        │
│  (Distributed Tracing / OpenTelemetry Trailing / Span Hierarchy)      │
└──────────────────────────────────┬────────────────────────────────────┘
                                   │
                                   ▼
┌───────────────────────────────────────────────────────────────────────┐
│       Secure Cryptographic Audit Trailing & Retention Logs            │
│  (Persisted Cryptographically to Immutable Storage - Encrypted AES)   │
└───────────────────────────────────────────────────────────────────────┘
```

### أ. القياس والتتبع والمسارات المتشعبة (Distributed Tracing Specification):
* جميع مكونات النظام وحزم بوابة التطبيقات ونطاقات قاعدة البيانات والعمال اللاتزامنيين تلتزم بتغذية مصفوفة البيانات المعيارية وتحديد خط الهوية `Correlation ID` مع بقاء الهياكل المتشابكة تحت مروحة **OpenTelemetry** لقياس سرعة قنوات الاستجابة وعزل نقاط الاختناق في أزمنة الاتصال في غمار العمليات الكثيفة.

### ب. سجل مراجعة البيانات التاريخي والحظر الجنائي للتكامل (Immutable Audit Trail Log):
* يلتزم النظام وبأنبوبة دفع برمجية عمياء ومستقلة تماما عن محرك البيانات اليومي بحفظ وتخزين أي حركة معاملة أو تعديل لخصائص الصلاحيات أو بيانات الإفرازات الحساسة وسر العمليات متبوعة بتفاصيل:
  * هوية الشركاء والفاعل `user_id + tenant_id`.
  * المعرف الفريد للطلب ومصدر الاتصال وحقيبة التتبع `IP Address + Correlation ID`.
  * الصورة الحرفية المتكاملة للقمة المشكلة للسطر قبل التثبيت وبعد الإجراء `Before State / After Value` مشفرة بأقوى معايير التشفير الصلبة (**AES-256**) ومعزولة في مستودع مستقل غير قابل للتلاعب به لحضور الفحص القضائي والتدقيق التنظيمي والتشريعي الموجه دوليا.

## 11. استراتيجيات وبناء قنوات القراءة وعزل قواعد التقارير (CQRS Read Model Engine)

لضمان عدم حدوث تدهور لسرعة إدراج القيود المحاسبية بالاستعلامات والفرز المطول للتقارير الربع السنوية الكبرى، يفصل النظام حيز القراءة تماما عن حيز الكتابة.

1. **دورة حياة حارس القراءات وإعادة البناء الذاتي (Projection Engine Lifecycle):**
   تدار سجلات ونماذج الإسقاط للقراءات عبر مستهلكين (Consumers) مخصصين يراقبون الأحداث الحركية ويعيدون صياغة الجداول الجاهزة المفرغة لتصدير قراءة المستعلم في Firestore أو مستودعات الكاش اللحظية في أزمنة استجابة قياسية (`Read SLA < 200 Milliseconds`).
2. **سياسات إدارة الذاكرة المؤقتة وإنهاك الكاش لضمان تكامل الأرقام ماليّا:**
   دعم تطبيق التصفية والمسح الفوري لعناقيد الكاش الرديف والتنظيم المتوازي لبيانات الإغلاقات عند حدوث تغير حركي في نبض البيانات المركزية المعتمدة لـ PostgreSQL.

## 12. إصدارات النظام وإدارة مسارات التحديث والتوافق (Evolution Strategy)

يتوطد استقرار المنصات متعددة الشركات بضمان سلامة مرور البنية مع تغير الإصدارات الفنية المجهرية وسقوط الترقيات.

```
       ┌────────────────────────────────────────────────────────┐
       │                Deployment Flow Control                 │
       │       (Feature Flags & Tenant Groups Management)       │
       └────────────────────────────────────────────────────────┘
                                   │
                  ┌────────────────┴────────────────┐
                  ▼                                 ▼
       ┌───────────────────────┐         ┌───────────────────────┐
       │     Canary Group      │         │     Primary Fleet     │
       │   (1% of Tenants)     │         │   (99% of Tenants)    │
       └───────────────────────┘         └───────────────────────┘
```

1. **إصدارات الأحداث بغير تدمير التوافق (Semantic Event Versioning):**
   * تلتزم كافة الرسائل وبنية الأحداث باتباع ميثاق إصدارات دلالية صارمة (Semantic Versioning: v1.0, v1.1...).
   * يحظر كسر التوافق العكسي للأحداث بإسقاط حقول رئيسية. عند إحداث تغييرات جذرية بكيان الرسالة، يلتزم النطاق بنشر حدث برقم إصدار ترقيمي جديد (e.g., `OrderApproved_V2`) وتظل المعالجات القديمة تدعم الإصدار السالف لفترة زمنية محكمة لتخليص الترحيل للـ (Consumers) كافة.
2. **سياسات النشر والتحول التدريجي المؤمن (Canary & Blue-Green Deployments):**
   دعم تخصيص خيوط مستهدفة لتشغيل واختبار الإصدارات الحديثة والترقيات بمرونة على كوكبة ضيقة ومحدودة من الشركاء والعملاء التجريبيين أولا (Canary Target: 1% of Tenants) ورصد استقرار ومؤشرات اتساق المحاسبة بالـ System وعقد الموازنة قبل التثبيت وتعميم الإصدار لكوكبة النظام الكبرى.

## 13. مخططات تدفق ومعالجة مسار البيانات والأحداث الهضمي (E2E Data Flow Mechanics)

ميثاق تتبع حيازة البيانات والترتيب الوعائي للتدفقات عبر الكيانات في لحظات الرخاء والتشوه الشبكي.

### مسودة حركة تشغيل مالية ومحاذاة تراجع فاشل (Financial Saga Step Walkthrough):

```
API Gateway (Validation Checked)
   │
   ▼
[Domain Services Layer]
   │  (Runs standard Unit of Work Transaction)
   │  (Step 1: Save Business Entity to Table)
   │  (Step 2: Save Integration Event to Outbox)
   │
   ├── Commit Transaction atomically in PostgreSQL SSoT ────┐
   │                                                         │
   ▼                                                         ▼
[Outbox Relayer Worker]                          [PostgreSQL Database]
   │ (Pulls Event from DB Outbox)                   │ (Local RLS Applied)
   ▼                                                │ (Bitemporal Logs Written)
Kafka Topic Partition (tenant_id Keyed)             └─────────────────────────┘
   │
   ├───────────────► [Accounting Engine Consumer] (Reads Event)
   │                        │
   │                        ├── Balancing Validation Approved?
   │                        │          │ (YES)
   │                        │          ▼
   │                        │   Writes immutable General Ledger Entries
   │                        │   Emits Event: LedgerPosted
   │                        │          │
   │                        │          └─► [Projection Worker]
   │                        │                 │
   │                        │                 ▼
   │                        │              Update Firestore (UI Presentation)
   │                        │
   │                        ▼ (NO - Balancing/Tenant validation failed)
   │                [Triggers Compensating Saga Reversal Flow]
   │                        │
   │                        ▼
   │                Rollback Business States (Send Compensating Command)
   │                Post Reversal Balance Sheet Adjustment
   │                Raise Instant Severity Compliance Alert
```

## 14. قواعد وحدود النظام المعمارية الحديدية الصارمة (System Hard Rules)

هذه القوانين والحدود غير قابلة للنقاش أو الإخلال بها تحت أي مبرر تجاري أو تفضيل تيسيري للمطورين. أي مخالفة لهذه الخيوط يجهض صلاحية وتكامل البنية بالكامل:

1. **PostgreSQL هو الإله الفني الحارس للصدق والحقيقة المالية:** لا يجوز اتخاذ أي مرجع رقمي أو تدوين مالي خارجي في Firestore أو الكاش كمرجع رسمي للحسابات.
2. **منع التعديل والحذف بالكلية بعد التثبيت المالي:** لا تدمير لسطور الأرصدة العامة واليومية؛ التعديل بالتسويات المحاسبية المتوازنة العكسية فقط.
3. **عقد العزل وحظر الولوج المزدوج المباشر للأوعية:** يمنع منعًا باتًا لأي نطاق برمجية تشغيلية الاستفسار أو مس السطور من جداول نطاق مجاور مباشرة. التداول يتم حصريًا بمذكرات التكامل الآمن وعقود API المربوطة.
4. **التحقق من العزل في مستودع البيانات الخادم:** لا تترك مسؤولية فلترة والتحقق من صلاحيات tenant المعتمد للمتصفح أو كود واجهة التطبيق البسيطة؛ عزل RLS في PostgreSQL يمثل جدار الفصل النهائي المستبسل حراسةً للأرقام.
5. **منطق الأعمال يعيش حصرياً في النطاقات الحركية:** يمنع منعًا باتًا وجود أي منطق لحساب الضرائب أو تسعير الصفقات أو التحقق من القوانين والائتمانات في كود الواجهة الأمامية أو جدران بوابة BFF.
6. **فيس دائم وبقاء لخطوط التتبع والتدقيق الجنائي:** لا معبر لطلب في ثنايا البنى التحتية للمنظومة لا ترافقه الهوية الموشومة `Correlation ID` مع بقاء كل سطر مراجعة وتصفية مشفرًا بالكامل.

## 15. المواصفات الفنية غير الوظيفية ومقاييس البناء والامتثال (NFR Matrix)

ولضمان خضوع البنية لرقابة الجودة التامة ومعايير الأمان الدولية، تلتزم البنية بقياسات محكمة تفرض هيبتها مع التوسع:

* **التوفر والموثوقية المطلقة (High Availability SLA):**
  الالتزام بمعدل تشغيل فعلي دائم مستمر بقيمة **99.95% Uptime** بالتضامن مع خطط النسخ المباشر اللحظي المتناغم والأرشفة التلقائية الحرة وتداول الأحمال المستمر.
* **سرعة ومستويات الاستجابة وعصب العمل (Performance Benchmarks):**
  * معالجة واعتماد وحفظ سطر القيد المحاسبي والترحيل تجتاز البوابة وتعتمد بالـ SSoT في زمن قياسي يقل تماماً وبنحو حازم عن **< 150ms**.
  * تقديم نتائج تقصي فحص التقارير والأرصدة العابرة للسنوات في طبقة الإسقاط والفرز لا يتخطى سداده حاجز الـ **< 1.5 Seconds**.
* **معدلات وأحجام الاستيعاب للتراكم اللحظي (Scalability Milestones):**
  بنيان الأنظمة تم تخطيطه معنويًا وهندسيًا لدعم تداول وإدارة **10,000+ Tenants** بالتوازي، واستيعاب كثافة وضخ مبيعات تفوق حاجز الـ **1,000,000+ Daily Financial Transactions** دون فقد متجانس للأداء أو التأثير اللحظي لسرعة السير بالمنصات.
* **مستويات سلامة واستعادة الكوارث وتكامل الأرصدة المالية (RPO & RTO Control):**
  * **RPO (Recovery Point Objective):** أقصى وقت مقبول لفقد تتابع الحركات المالية تحت أثر سقوط مدمر هو **RPO = 0 Seconds** (لا للتنازل عن أي صف لمحطة مأكولة ماليًا، لالتزامنا بالـ Serialized Transactions وقوة النسخ المتزامن اللحظي للعمود الفقري بـ PostgreSQL)، وللمعاملات التشغيلية المعتادة **RPO < 5 Minutes**.
  * **RTO (Recovery Time Objective):** أقصى فترة زمنية مباحة لإنعاش وبث تيار العمل للأنومة الساقطة وتسييس الماكينات من جديد للمستعملين هي **RTO < 30 Minutes** استنادًا للمركبات المجمعة وقوة الجاهزية.

### وثيقة ميثاق الحوكمة المعمارية - مراجعة واعتماد:
* **حالة الاعتماد للبنية:** تم الحفر والتثبيت كـ ميثاق ملزم وغير قابل للتمرير دون الخيار لبرمجيات الاستقرار في دورة التطبيقات وتكامل الدعم.
* **حارس الجودة والتفتيش البرمجي (Fitness Functions CI/CD Checks):** تفتش دوال الحصانة المحبوسة بالـ Pipeline في كل التزام لإلقاء أي بناء يحاول تجاوز عتبة الفصل أو سكب المنطق خارج النطاق المغلق.
